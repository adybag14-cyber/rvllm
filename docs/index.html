Abstract

We present rvLLM, a from-scratch Rust implementation of the vLLM inference engine for OpenAI-compatible API serving with CUDA GPU acceleration. The system comprises 23 Rust crates and 15 pre-compiled CUDA PTX kernels, compiling to a single 16 MB static binary. On a single NVIDIA B200 (180 GB), rvLLM achieves 3,946 tok/s serving Qwen2.5-1.5B with zero errors across 13,553 verified requests at batch sizes up to N=4,096. On a single A100 SXM4 (40 GB), it reaches 3,191 tok/s -- 16x faster than Python vLLM 0.18 on identical hardware. This speedup derives from three reinforcing mechanisms: (1) GPU-resident greedy decoding via an argmax kernel that reduces device-to-host transfer by ~32,000x per token, (2) CUDA graph replay that collapses the entire decode forward pass into a single driver call, and (3) a zero-copy memory architecture with no Python objects, no PyTorch tensor metadata, and no garbage collector. Startup time is 6--9 seconds versus 121 seconds for Python vLLM, CPU RSS is 348 MB versus 1,033 MB, and the binary is 31x smaller.

1. Introduction

Large language model inference serving has become critical infrastructure. The dominant open-source engine, vLLM, introduced PagedAttention for efficient KV cache management. However, vLLM is implemented in Python with PyTorch, which introduces systemic bottlenecks:

  • Python's GIL serialises the scheduler, tokenizer, and output processing. At high concurrency, the scheduling loop itself becomes the bottleneck.
  • Garbage collection pauses grow as Python tracks millions of tensor metadata objects. Each torch.Tensor carries ~50 bytes of overhead.
  • Deployment complexity: ~500 MB of dependencies, 30--120 second startup for imports, Triton JIT, and NCCL init.
  • The device-to-host bottleneck: after the GPU forward pass, the entire logit matrix is transferred to host for Python-side sampling. For Qwen2.5-1.5B (vocab 151,936) at batch 128, that's 74 MB per decode step.

rvLLM addresses all of these. A single 16 MB binary with zero runtime dependencies. Rust's ownership model for deterministic deallocation. Direct cuBLAS and CUDA kernel invocation. And a GPU-side argmax kernel that reduces the dominant DtoH transfer from 74 MB to 512 bytes for a 128-sequence greedy decode step -- a ~150,000x reduction.

2.1 The Forward Pass

All activations remain on GPU throughout the entire layer loop. The execution flow per step:

  1. HtoD upload of metadata: token IDs, positions, context lengths, block tables, slot mappings, and seq_start_pos (cumulative query-length prefix sums for mixed prefill+decode).
  2. Embedding lookup via embedding_gather_kernel (GPU-resident).
  3. N transformer layers: RMSNorm, Q/K/V projections (cuBLAS), rotary embedding, reshape-and-cache (scatter K/V into paged cache), FlashAttention-2, output projection, residual + post-norm, gate/up projections, SiLU activation, down projection, second residual.
  4. Final RMSNorm and LM head projection (cuBLAS).
  5. Greedy fast path: GPU argmax kernel returns one i32 per sequence. Non-greedy: full logit transfer to host for CPU sampling.

2.2 Mixed Prefill+Decode Batching

Prefill sequences (q_len > 1) and decode sequences (q_len = 1) coexist in the same GPU batch. The seq_start_pos array disambiguates them to the attention kernel. This eliminates the pipeline bubble between prefill and decode phases.

2.3 Memory Management

The entire KV cache is allocated as one cudaMalloc call at startup. Block allocation is a VecDeque::pop_front; deallocation is push_back. Zero per-request GPU allocation calls. Copy-on-write for beam search. LRU prefix caching for shared system prompts. Swap to pinned CPU memory when GPU blocks are exhausted.

3. CUDA Integration

3.1 The GPU Argmax Kernel

The single most impactful optimisation. Standard LLM serving transfers the entire logit matrix (batch x vocab x 4 bytes) from GPU to host every decode step. For Qwen2.5-1.5B at batch 128: 74 MB per step. The argmax kernel finds the max-probability token directly on GPU and returns only the winning token IDs: 512 bytes per step. A ~150,000x reduction in per-step DtoH transfer.

3.3 CUDA Graph Replay

After 3 warmup calls, decode steps are captured into CUDA graphs keyed by padded batch size {1, 2, 4, 8, 16, 32}. cuGraphLaunch replays the entire forward pass as a single driver call, eliminating ~100 us of per-kernel launch overhead per step.

4. Why rvLLM Uses Less Memory

The advantage is structural, not algorithmic. Same PagedAttention algorithm, different per-object overhead:

  • No Python objects: A KV block in Python vLLM is a dict with ~50 bytes overhead. In rvLLM it's a u32 ID + AtomicUsize ref count = 24 bytes in a pre-allocated VecDeque.
  • No PyTorch tensor metadata: Every torch.Tensor carries autograd state, strides, device info. rvLLM holds CudaSlice<f32> -- a pointer and length.
  • Single pre-allocated slab: One cudaMalloc at startup. No per-request GPU allocation.
  • Memory-mapped weight loading: memmap2 maps SafeTensors from disk. The OS pages on demand; the model is never fully in CPU RAM.
  • No NCCL buffers: Single-card inference eliminates tensor parallelism overhead (5--15% NVLink overhead + ~2--5 ms NCCL sync per step).

4.1 The Single-Card Advantage

A single B200 at 8 TB/s memory bandwidth reads 140 GB of a 70B model in 17.5 ms. With N=64 concurrent sequences, each 17.5 ms window produces 64 tokens = 3,657 tok/s. No multi-GPU communication overhead. This matches the empirical plateau we observe.

5.5 Where the Speed Comes From

  1. GPU-resident argmax: eliminates 74 MB DtoH transfer per step, saving ~3 ms at PCIe Gen4.
  2. CUDA graph replay: collapses N_layers x ~10 kernel launches into one cuGraphLaunch.
  3. Mixed prefill+decode batching: eliminates pipeline bubbles between phases.
  4. No GIL: scheduling, tokenisation, sampling run on parallel Rust threads via Rayon.
  5. No GC pauses: deterministic deallocation via Rust ownership.
  6. Zero-copy activation residency: all activations on GPU throughout the N-layer forward pass.
  7. Prefix caching: shared system prompts reuse computed KV blocks across requests.

7. Related Work

vLLM -- PagedAttention and continuous batching for LLM serving. rvLLM reimplements the architecture in Rust, eliminating GIL, GC, PyTorch tensor overhead, and adding GPU-side sampling.

TensorRT-LLM -- NVIDIA's optimised inference with operator fusion and hardware-specific kernels. Requires model-specific compilation; tied to proprietary runtime. rvLLM targets simpler deployment (single binary).

llama.cpp -- Efficient C/C++ inference with extensive quantisation and broad hardware support. Excels at single-user local inference. rvLLM targets multi-tenant serving with continuous batching and OpenAI-compatible API.

candle -- Rust ML framework from Hugging Face. Provides tensor primitives; rvLLM implements the full serving stack on top of direct cuBLAS calls.

8. Conclusion

rvLLM demonstrates that a systems-language reimplementation of LLM serving infrastructure can achieve 16--20x higher throughput than the Python incumbent while using 3x less CPU memory, starting 15--20x faster, and deploying as a 31x smaller binary. The system has been verified with zero errors across 14,620 requests at batch sizes from N=1 to N=4,096 on both B200 and A100 GPUs, with coherent text output confirmed at every scale.
