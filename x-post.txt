updated rvLLM and synced the public repo/docs/paper/site to the current H100 results.

current direct-engine comparison against vLLM 0.19.0 on the same H100, same Qwen2.5-7B snapshot, same output len:

- N=1: 152.9 tok/s vs 167.6
- N=64: 9589.1 tok/s vs 9256.4
- N=128: 15479.9 tok/s vs 16904.8

so the honest headline is:

- rvLLM now wins cleanly at N=64
- it is much closer at N=1 than before
- it still trails at N=128

the important architecture corrections were:

- the normal T=1 path is the reusable batched scratch path, not the old `CublasGemvDecode` story
- the batched GEMM policy is explicit instead of half-wired
- the real N=64 hotspot was Hopper FFN `gateup_silu -> down`, and the winning fix was a CUTLASS SM90 auxiliary-epilogue gate path

the batched side is still the same explicit hybrid policy:

- QKV / O / down on cuBLAS or cublasLt
- gate activation on CUTLASS SM90 aux epilogue

tooling mattered too:

- stable direct benchmark harness
- fixed `cublasLt` autotune-cache replay
- runner-side phase timing on the real decode bucket
- CUTLASS shared-library + Rust FFI loop for real H100 epilogue work

still true: stock vLLM has the stronger low-batch decode stack overall, and it is still ahead at N=128. but rvLLM now has a real public batched win instead of just "near parity".
